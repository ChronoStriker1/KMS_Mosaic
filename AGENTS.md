# AGENTS.md (Project Guidance)

The role of this file is to describe common mistakes and confusion points that agents might encounter as they work in this project. If you ever encounter something in the project that surprises you please alert the developer working with you and indicate this is the case in the AgentMD file to help prevent future agents from having the same issue.

## Surprises / gotchas
- Restarting `kms_mosaic` on the Unraid host is sensitive to stop/start ordering. Do not stop and start in parallel. `pkill -f` can also match the remote shell command line and race badly. Use an exact-name stop like `pkill -x kms_mosaic.bin`, wait briefly, then start the plugin service or wrapper.
- The active config file is now watched at runtime. A change triggers an in-process self-reexec with the original argv, so touching the config is enough to reload it and the PID stays the same.
- The old Unraid userscript can override runtime behavior independently of the repo. On this host it had `KMS_MOSAIC_DISABLE_CONFIG_WATCH=1`, which silently disabled live config reload even though the binary supported it. If a runtime feature mysteriously stops working, check whether the host is still launching through the old userscript instead of the plugin.
- Even with the correct sequential flow, a fast restart can still hit `drmModeAtomicCommit (modeset): Permission denied` on the Unraid host. If that happens, wait a few seconds and start it again.
- The plugin service stop path can hit wedged `kms_mosaic.bin` PIDs that ignore a normal `SIGTERM`. If restart appears to do nothing, check whether the old PID survived and use an exact-PID `SIGKILL` fallback before starting again.
- The containerized packaging path is the most reliable validation route on non-Linux hosts. If you need a Linux build locally, prefer `scripts/macos_build_pkg.sh` over the host `make` path.
- On the Unraid host, launching `/usr/local/bin/kms_mosaic.bin` directly can miss packaged shared-library resolution. Use the `/usr/local/bin/kms_mosaic` wrapper or the userscript path, not the bare binary, when validating remotely.
- The package wrapper now exports `KMS_MOSAIC_REEXEC` so config-watch reloads come back up through `/usr/local/bin/kms_mosaic` instead of reexecing the bare binary. If plugin-managed startup loops and then dies on `libmpv.so.1`, check that the wrapper and packaged binary were rebuilt together.
- The native Unraid plugin now lives under `unraid-plugin/` and is built with `scripts/build_unraid_plugin.sh`. It depends on the Linux package already existing in `dist/`.
- The plugin-owned web service must be started through `/usr/local/bin/kms_mosaic_web`, not by executing `kms_mosaic_web.py` directly, because the wrapper selects the Python venv that has `aiortc`/PyAV available.
- The Unraid plugin page no longer embeds the editor through an iframe. It now injects the editor directly into the plugin page, sources the HTML and config state from the packaged `kms_mosaic_web` wrapper, and only uses the web service for preview/media/WebRTC endpoints. If the plugin UI breaks while the standalone editor still works, check the wrapper-backed actions in `include/actions.php` and the HTML/JS rewrite path in `KMSMosaic.page` first.
- The current intended plugin route is the Settings-side `KMSMosaic` page, but the page itself must still behave like a full standalone plugin surface, not like two separate settings pages bolted together. Keep the service controls folded into the main page and preserve a normal plugin-launch experience from the Plugins tab.
- If the plugin web service looks alive in the browser but reports `web_running=false`, treat stale PID-file handling as a first suspect. The service script now falls back to `pgrep` on the `kms_mosaic_web.py --config ... --port ...` command line.
- The plugin install path now retires the old `Start kms_mosaic` userscript by renaming it to `Start kms_mosaic.disabled-by-kms-mosaic-plugin`. If boot ownership seems inconsistent after plugin install or removal, check both paths.
- If the Settings page keeps serving an older `KMSMosaic.page` after reinstall, check for duplicate copies anywhere under `/usr/local/emhttp/plugins` first. Older bad bundles left nested trees like `/usr/local/emhttp/plugins/usr/local/emhttp/plugins/kms.mosaic/KMSMosaic.page`, and even manual backups such as `/usr/local/emhttp/plugins/kms.mosaic.backup/KMSMosaic.page` can override the real page. Move stale copies fully out of `/usr/local/emhttp/plugins`; renaming them in place is not enough.
- The repository architecture described in older docs may be stale. The current compositor is already split across `app`, `display`, `media`, `render_gl`, `frame`, `panes`, `layout`, `options`, `runtime`, and `ui` modules.
- The compositor’s live runtime, option parsing, and layout output now allocate pane/role storage dynamically from `pane_count`. The main remaining legacy assumptions are mostly user-facing naming/help shortcuts (`--pane-a` through `--pane-d`) and related labels, not fixed-cap live buffers.
- The companion web UI is now a tree-first editor rather than just a static form. It saves `--split-tree` specs, demotes legacy scene/raw config fields into Advanced sections, and uses a WebRTC preview path instead of the older browser-side BMP/snapshot loop.
- The pane inspector now has structured per-pane mpv controls layered on top of pane-local raw mpv options. If you add another pane-local mpv setting, update both the structured parser/builder in `tools/kms_mosaic_web.py` and the raw fallback textarea handling so they stay in sync.
- The web API's `/api/state` response is now shaped as `{ config_path, raw_config, state }`. If you smoke-check it manually, don't expect pane/layout fields at the top level.
- Compositor preview files are still written via temp-file rename before the WebRTC bridge consumes them. If the live preview starts showing torn frames again, treat that as a compositor-side regression first, not just a browser/UI bug.
- `tools/kms_mosaic_web.py` may have live user edits that have not been committed yet. Check the local diff before changing it and patch surgically instead of assuming the repo version is authoritative.
- The current Unraid web-preview venv has AV1 encoders available through PyAV, but `aiortc` still only advertises `VP8`, `H264`, and the locally patched `VP9` codec layer. Better CPU headroom does not make AV1 available by itself; adding AV1 requires codec-layer support in `aiortc`, not just encoder presence in FFmpeg/PyAV.
- A failed or partial plugin install can leave a zero-byte cached plugin bundle under `/boot/config/plugins/kms.mosaic/`. If the Unraid plugin extractor starts failing unexpectedly, verify the cached `.tgz` on the host before blaming the install script.
- The remaining explicit user follow-ups are now narrower: improve the preview toward a truer 1:1 mirror, add direct drag/resize handles in the Layout Studio, and expose more common mpv options as structured controls so fewer edits need the raw advanced text.
- The plugin-hosted settings page must not reuse DOM ids from the embedded editor. Reusing ids like saveBtn/reloadBtn makes the embedded JS bind to the wrong controls and silently breaks config save/reload from the page.
- If playback starts stuttering intermittently after reloads, inspect the live process tree for orphaned pane children first. Older builds could leave `btop`, `tail`, or other pane commands running under `init`, which adds random extra load even though the current pane tree looks normal.
- Multi-mpv-pane playback needs `mpv_render_context_report_swap()` called for every pane-local mpv context after present, not just the primary video context. If pane-local video gets strangely choppy while CPU looks fine, inspect `src/frame.c` before blaming decode settings.
- Multi-pane mpv rendering should not share one scratch video FBO across differently sized panes. That can trigger repeated texture/FBO reallocations, intermittent GPU stalls, and big RSS swings even when the config is unchanged.
